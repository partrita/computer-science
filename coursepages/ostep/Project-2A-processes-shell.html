치트 팁 (Tips)

  • 유닉스 셸과 관련된 토론 3번 비디오를 꼭 시청하십시오.
  • OSTEP 교재의 챕터 5를 정독하십시오.
  • 첫 시작은 아주 초라하게 구현하세요: 프롬프트를 띄우고 사용자가 exit를 입력하면 얌전히 종료되는, 오직 한 가지 기능만 가진 껍데기 셸부터 짜는 겁니다. 그다음 cd 기능을 덧붙이고, 그 뒤 path 기능을 얹으세요. 그다음에야 비로소 execv를 통한 일반 커맨드 실행 로직을 넣고, 이어서 일괄 처리(batch) 모드, 리다이렉션, 마지막으로 병렬 처리 로직 순으로 하나하나 층을 쌓아 올리며 접근하세요.
  • 테스트 스크립트 봇들은 대부분 일괄 처리 모드와 리다이렉션 방식을 통해 점검을 수행할 테니, 여러분이 이 기능들을 구현해내기 전까지는 여러분 스스로가 셸을 수동으로 두들겨보며 눈으로 테스트해야만 합니다.
  • path 커맨드를 짤 때는, 시스템의 절대 경로와 상대 경로 모두를 포용할 수 있게 설계해야만 합니다 (예: path tests와 같이 상대적인 것이든 path /usr/bin같이 절대적인 것이든 다 소화해야 합니다).
  • 에러 로그들을 정교하게 타이밍 맞춰 뿌리는 작업은 꽤나 골치 아픕니다. 초기엔 일단 합당해 보이는 모든 틈새마다 닥치는 대로 대충 에러 메시지를 발사하게 짠 뒤, 테스트 스크립트를 돌려 피드백을 맞아가며 정답 에러 로그가 터져야 할 바로 그 정확한 임계점에만 에러 코드가 나오도록 코드를 깎아내가시면 됩니다. 만일 i번 테스트를 돌리는 중이라면, tests/i.err와 tests/i.rc를 까보면 셸이 몇 번의 에러를 뱉어내야 하는지 정답이 다 적혀 있습니다. 이를 눈으로 여러분의 산출물인 tests-out/i.err, tests-out/i.rc와 쉴 새 없이 스캔하며 대조해 보면 됩니다.
  • 테스트 3번을 돌리던 중, 테스트 봇은 ls: cannot access ... 꼴을 간절히 바라마지 않는데 여러분의 셸은 영양가 없이 길게 /bin/ls: cannot access ... 나 /usr/bin/ls: cannot access ... 같이 지저분하게 출력해서 퇴짜를 맞는다면, 여러분의 냅킨만 한 $PATH 환경 변수의 선두 자리에 억지로 /bin 을 처박아 보세요. 그래도 약발이 안 먹힌다면, 그냥 쿨하게 tests/3.err 원본 정답 파일 텍스트 자체를 여러분 시스템이 내뿜는 로그 모양새와 일치되도록 뜯어고쳐 버리세요. 애초에 시스템 구동부의 출력 모양새는 ls 나 execv 자체 시스템 소스를 갈아엎지 않는 이상 제어가 안 되는 불가항력이므로, 기능의 정신만 제대로 동작하고 있다면 이런 자질구레한 텍스트 따위로 인한 불합격은 그냥 조작해 무시하고 지나가도 대세에 전혀 지장 없습니다.
  • 저 역시도 시스템 세팅 덕에 테스트 통과를 위해서 /tests/3.pre 내용 안에 강제로 /bin/ls 경로 사용을 지시하도록 파일을 뜯어고쳐야만 했습니다. 이게 영 찝찝하시다면 아예 시스템 셸 .profile 이나 .bashrc 자체에 export PATH="/bin:$PATH" 구문 한 줄을 처박아 두시면 만사형통입니다.

메모리 장난질의 치명적 함정과 지뢰들 (Memory Management Traps and Pitfalls)

  • 이 미션은 아주 손금 보듯 투명하게 뻔한 삽질을 유도합니다. 생명주기가 끝나면 아스라지게 사라져버릴 스택(stack) 메모리 안의 변수를 가리키는 위험천만한 포인터를 무심결에 만들도록 유혹하며, 그 즉시 무자비한 자비 없는 세그먼테이션 폴트(segmentation fault) 크래시로 당신을 응징할 것입니다. 포인터로 어떤 문자열을 건들고 싶다면 그 문자열 덩어리가 스택 따위가 아니라 반드시 든든한 힙(heap) 공간에 이쁘게 할당된 놈인지 목숨 걸고 확인하십시오.
  • 이 피의 조언에도 불구하고 기어코 스택에 있는 문자열 따위를 당겨 써야겠다면, 반드시 힙 공간에 새롭게 할당한 문자열 방앗간을 마련하고 거기로 strcpy(), strncpy(), strcat(), strncat() 일꾼들을 시켜 안전하게 알맹이만 복사해 넘겨 써야 합니다.
  • 오직 크기가 견고하게 명확히 확정된 문자열 조합에 대해서만 strcpy()와 strcat() 을 쓰도록 허락합니다. 이때 복사를 받을 그릇 공간 사이즈가 원본 문자열을 모두 토해내고도 마감재 역할인 \0 하나가 들어갈 여유 공간 한 칸을 더 남겨둘 크기인지 반드시 셈하십시오.
  • strncpy()와 strncat() 에 한해, 파라미터 n 값이 \0 문자가 탑승하기에 충분히 큰 크기를 보장하는지 두 번 보고, 아니면 수동으로 강제로 끄트머리에 구겨 넣으십시오.
  • 쥐도 새도 모르게 쓰고 또 쓰는 좀비, 즉 ‘해제 후 사용 (use-after-frees)’ 버그 사태를 경계하십시오, 이 악마는 특히 path 개발 중 지독하게 엿을 먹일 것입니다.
  • getline()이나 strdup() 등을 거쳐 만들어진 문자열 잔해물들은 전부 깨끗하게 해제(free)하여 치우십시오. 다만 한 번 해제한 놈을 또 날려 보내려 드는 이중-해제(double-frees)의 바보짓은 경계하십시오, 예컨대 이미 저세상으로 증발시킨 문자열 부스러기 파편들을 한 번 더 증발시키려는 미치광이 짓 말입니다.
  • C 라이브러리의 함수 중 strtok() 같은 함정 픽은 무조건 기피하십시오; 애초에 스레드 세이프(thread-safe)라고는 눈 씻고도 찾아볼 수 없는 악마입니다. 대타로 무조건 strsep()을 기용하십시오.
  • strsep()을 꺼내 든 순간, 조각나기 전의 원본 문자열 포인터 사본을 따로 주머니에 고이 보관해야만 나중에 메모리 해제(free)가 가능해집니다. 왜냐하면 strsep() 이놈은 자르면서 원본 포인터 위치를 마구 변형시키고 파괴해 놓기 때문입니다. 그 오염된 포인터를 들고 메모리를 해제하려 들면 귀하의 페이지 테이블 뚝배기가 깨져버릴 것입니다.
  • 당연하게도 strsep(&buf, delim)을 때린 직후엔, buf 놈이 NULL 빈 껍데기가 되어있는 건 아닌지 확인 전까진 절대 해제시켜 작동하려 들지 마십시오.
  • C를 짜는 일반론적 진리: 당신이 혹여 특정 함수 지붕 아래서 데이터를 위해 메모리를 할당(allocate)받아 냈다면, 쓴 뒤 해제(free)하는 짓 역시 무조건 그 함수 지붕 아래서 책임을 져야 합니다. 만약 코드를 예쁘게 만든답시고 특정 create_xxx 같은 껍데기 함수에서 메모리를 띄웠다면 반드시 짝꿍으로 그 오물들을 치울 destroy_xxx 해제 전담 함수도 구비시켜야 합니다. 즉 띄움과 던짐의 층위 격차는 항상 평행해야 하며, 이 철칙이 메모리 지뢰를 안 밟을 가장 핵심 방탄복이라는 뜻입니다.
  • malloc, calloc, 혹은 realloc 부류들을 호출해 냈을 땐 뒤이어 반드시 반환값이 재수없게 NULL은 아닌지 색출하십시오.
  • 포인터 나열이 들어간 배열체 따위를 생산할 심산이라면 빈 찌꺼기 포인터 값이 침범하는 걸 차단하기 위해 malloc 대신 calloc을 쓰는 걸 추천합니다.
  • 여담으로 update_path 로직 안에서 골치 아팠던 일을 전해보자면: 수많은 테스트 스크립트들이 path /bin /usr/bin 요딴 식으로 멀쩡하게 들어오는데 갑자기 튀는 놈 딱 하나가 path tests 이따위로 경로를 박아 넣는 패악질을 쳤습니다. 그래서 전 화끈하게 꼼수를 부려: 텍스트 시작 꼬리말이 무조건 슬래시(/)부터 시작하면 ‘그래 넌 절대경로구나’ 치고 원본 그대로 밀어 넣어 복사해 버리고, 그게 아니라면 ‘어 이놈 상대경로구나’ 판단하여 문자열 앞대가리에 무조건 ./ 두 마디를 때려 박아 강제로 정돈해 버렸습니다.
