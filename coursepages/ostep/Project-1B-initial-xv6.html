프로젝트 1B (Project 1B)

모든 감사를 palladian 님께 전합니다

리눅스 설치 (Linux Installation)

  • 호환되는 컴파일러 툴체인이 있는지 확인하세요. 리눅스 환경이라면 gcc가 완벽하게 작동할 것입니다.
  • qemu-system-x86를 설치하세요 (시스템에 따라 qemu-system-i386 또는 qemu-system-x86_64로 불릴 수 있습니다. 일부 배포판에서는 qemu 자체가 잘못된 패키지일 수 있으니 주의하세요).
  • Perl을 설치하세요.
  • gawk를 설치하세요.
  • expect를 설치하세요.
  • 프로젝트의 테스트 스크립트와 동일한 디렉토리에 src/ 디렉토리를 만드세요.
  • xv6 깃허브 저장소를 클론하고 소스 코드를 src/ 디렉토리에 복사하세요.
  • src/ 디렉토리 안에서 make qemu-nox를 실행하여 xv6가 정상적으로 작동하는지 테스트하세요. Ctrl-a 누른 후 x를 눌러 xv6를 종료할 수 있습니다. 이를 깜빡했다면 qemu 프로세스를 강제로 종료해도 됩니다. top이나 htop을 확인하여 qemu가 확실히 종료되었는지 점검하는 편이 좋습니다. 가끔 종료 후에도 백그라운드에서 계속 실행되며 시스템 자원을 소모할 수도 있기 때문입니다.
  • Makefile을 수정하여 CPUS := 1로 설정하세요.
  • 다시 make qemu-nox를 실행하여 xv6가 여전히 잘 돌아가는지 테스트하세요.

지시 사항 (Instructions)

  • 여러분의 임무는 xv6에 새로운 시스템 콜인 getreadcount()를 생성하는 것입니다. 이 시스템 콜은 이전에 발생했던 총 read 시스템 콜의 호출 횟수를 반환해야 합니다. 주의할 점은 이 카운트가 개별 프로세스의 카운트가 아니라 시스템 전체의 전역적(global)인 카운트여야 한다는 것입니다.

권장되는 접근 방식 (Suggested Approach)

  • xv6 소스 코드 PDF를 다운로드하세요 (다운로드한 실제 소스 파일들보다 PDF 상에서 훨씬 정리가 잘 되어 있습니다). 목차를 읽어보고 시트, 페이지 및 줄 번호가 어떻게 매겨져 있는지 파악하세요. 그 후 뒤쪽에 나오는 상호 참조(cross-references) 목록을 훑어보아 나중에 코드의 특정 부분을 찾아야 할 때 어떻게 탐색할 수 있는지 알아두세요.
  • 목차에서 processes와 system calls 영역, 그리고 user-level 섹션 아래의 usys.S 파일에 해당하는 xv6 소스 코드 부분들을 (매우) 빠르게 훑어보세요. 아직 완벽히 이해하려 들지 않아도 됩니다. 나중에 교수님의 해설 비디오와 여러분의 환경 구조상 디렉토리 배치가 다를 것이기 때문에, 그저 각 파일이 PDF의 어느 위치쯤에 존재하는지 눈으로 찍어두는 용도일 뿐입니다.
  • 프로젝트 P1B를 다룬 2번 토론(discussion) 비디오를 시청하면서, 다운로드한 xv6 코드 PDF 사본을 눈으로 쫓아가세요. 교수님이 각 파트의 기능을 설명할 때 PDF 위에 직접 주석을 달거나 필기해 두는 것이 좋습니다.
  • 프로젝트의 깃허브 페이지에 링크된 배경 지식 섹션을 읽고, 역시 xv6 코드 PDF에 메모를 남기세요.
  • 다시 한번 xv6 PDF를 끝까지 통독하세요. 이번에는 processes와 system calls 섹션들, 그리고 usys.S 및 user.h 파일의 전반적인 개념을 이해하는 데 집중하세요 (참고: user.h 파일은 xv6 PDF에 포함되어 있지 않으므로, 여러분이 실제로 다운로드받은 코드를 열어서 확인해야 합니다). 이 역시 코드 한 줄 한 줄을 전부 완벽하게 분석하려 얽매이지 말고, 시스템 콜이 어디서 정의되고 어떤 방식으로 호출되는지 등의 큰 뼈대를 파악하는 것으로 충분합니다.
  • xv6 소스 코드를 수정하여 새로운 getreadcount() 시스템 콜을 추가하세요. 필연적으로 여러 개의 파일들을 건드려야 할 것입니다. 나중에 디버깅할 때 수정 지점을 한눈에 찾기 쉽도록 모든 변경 사항 옆에 // OSTEP project라는 주석 표식을 달아두기를 강력히 추천합니다.
  • 여태 본론으로 말하지 않았던 한 곳, xv6 PDF에는 누락되어 있으나 반드시 코드를 손봐야 할 부분이 바로 user.h 파일입니다.
  • 코딩을 마쳤다면 테스트 스크립트를 실행시켜 보세요. 테스트 1번은 read 콜을 여러 번 발생시키는 함수를 먼저 돌린 뒤, getreadcount를 호출합니다. 여러분의 시스템 콜이 제대로 설계되었다면, 모든 프로세스 전체에서 발생한 read 콜의 누적 합계를 완벽히 추적해 내야만 합니다.
  • 만일 여러분의 코드가 테스트 1번을 통과했다면 진심으로 축하합니다! 일단 여기까진 목표 달성입니다. 테스트 2번에 대한 건 동시성(concurrency) 렉처를 모두 수강한 후에 고민해도 늦지 않으니 지금은 무시하세요.
  • 만약 테스트 1번에서 막혔다면 tests/1.out 파일의 기댓값과 여러분이 출력한 실제 결과값인 tests-out/1.out을 서로 대조하고 추적하십시오. 덤으로 tests-out/1.err 파일을 들여다보아 에러 로그가 남지 않았는지도 조사해 보아야 할 것입니다.
  • 여러분 스스로가 프롬프트 터미널 창에서 직접 make qemu-nox를 입력하고 xv6 환경 안에 진입해 테스트해 볼 수도 있습니다. 시스템 진입 후 ls 명령어로 파일들을 조회하면 test_1과 test_2라는 실행 파일이 보일 겁니다. ./test_1을 타이핑해 테스트 1번을 수동으로 작동시킨 후 직접 눈으로 튀어나오는 출력물을 확인하여 기댓값과 맞춰볼 수도 있습니다.
  • 마침내 스레드(threads), 동시성(concurrency), 락(locks)에 관한 렉처를 모두 시청하셨다면: 이제 테스트 2번 차례입니다. 테스트 2번은 여러분이 짜놓은 getreadcount가 스레드 안전성(thread-safe)을 보장하는지 검사합니다. 아마 락 처리 기능을 추가하기 전까진 안전하지 않은 상태일 테니 락(lock) 구문을 집어넣고 코드를 수정해야만 합니다. 그런 다음, 다시 테스트 스크립트를 가동하여 코드가 1번 2번 테스트 모두를 완벽히 통과하는지 확인하십시오.
